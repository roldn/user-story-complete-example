## 1. What is a CRUD / ABM?

**CRUD** stands for **Create, Read, Update, Delete** — the four fundamental operations that can be performed on a set of records in a system.

In Latin American software development (particularly Argentina), the equivalent term is **ABM**, which stands for **Alta, Baja y Modificación** (literally: Register, Deactivate, Modify). Both terms describe the same concept.

| ABM Operation | CRUD Equivalent | Description |
|---|---|---|
| **Alta** (Register) | Create | Add a new record |
| — | Read | View / list existing records |
| **Modificación** (Modify) | Update | Edit an existing record |
| **Baja** (Deactivate) | Delete | Remove or deactivate a record |

> In a business context, **Delete** is rarely a physical deletion (removing the row from the database). In most HR and people management systems, it is implemented as a **soft delete**: the record remains in the system but its status changes to "Inactive". This is critical for auditing, historical tracking, and legal compliance.

### Why an Employee CRUD?

An Employee CRUD is the foundational module of any Human Resource Management System (HRMS/HCM). Without it, the following are not possible:

- Processing payroll (the payroll system needs to know who the employees are)
- Managing building or system access
- Generating headcount reports for management
- Meeting legal reporting obligations with regulatory bodies such as AFIP/ARCA

In short, it is a **foundational module**: other modules depend on it existing and working correctly.

## 2. What is a User Story?

A **User Story** is a short, user-centered description of a feature a system should provide. It does not describe *how* to build it technically — it describes *what the user needs* and *why*.

### Base format

```
As a [type of user],
I want to [perform an action],
so that [I get a benefit / solve a problem].
```

This three-part format, known as the **Connextra template** (named after the company where it originated), forces the author to:

1. Identify **who** has the need (a person, not the system)
2. Define **what** they want to do specifically
3. Justify **the value** that action generates for them

User Stories are the basic unit of work in agile methodologies like **Scrum** and **Kanban**. They are intentionally simple at their core to encourage conversation between the Product Manager, the development team, and stakeholders.

## 3. Thinking Process

The following describes the reasoning that led to building this User Story, step by step.

### Step 1 — Understand the problem domain

The first step was identifying the context: an **Employee CRUD** within a business management system.

Guiding questions:

- Who uses this module? → Not only HR Admins; managers and employees themselves have distinct needs.
- What problem does it solve? → Replacing a manual, Excel-based process prone to errors and inconsistencies.
- What happens without it? → Other systems (payroll, access control) have no reliable data source.

### Step 2 — Identify all personas involved

A common mistake in user story design is writing from a single actor's point of view. In this case, the module has at least four profiles with different needs:

- **HR Admin**: needs to operate (create, deactivate, modify)
- **Area Manager**: needs to consult their team roster
- **Employee**: needs to view and validate their own data
- **Auditor**: needs a traceable change history

This directly shaped the **business rules** (who can do what) and the **acceptance criteria**.

### Step 3 — Define the domain data

Before thinking about screens or flows, you need to know what information the entity holds. The data was separated into three groups with their own logic:

- **Personal data**: belongs to the person; exists before the employment relationship
- **Employment data**: defines the relationship between the person and the company
- **System data**: generated by the application for its own functioning (employee ID, email, audit trail)

This separation also has **privacy and permissions implications**: not all roles should see all fields.

### Step 4 — Write acceptance criteria covering the happy path and edge cases

Acceptance criteria translate the story into verifiable conditions. The guiding principle applied:

> "If QA cannot write a test case from this criterion, the criterion is poorly written."

The following were covered:
- The **happy path** (successful creation, successful edit)
- **Edge cases** (duplicate National ID, missing required fields)
- **Query operations** (search, filtering, export)
- **Deactivation** as an operation with its own non-trivial logic

### Step 5 — Derive the business rules

Business rules are not acceptance criteria — they are restrictions and behaviors that apply across multiple criteria. They were extracted by asking:

- What if two employees have the same name? → The Employee ID differentiates them and is auto-generated.
- Can a manager deactivate someone from their team? → No, only HR Admin can.
- What happens to the data of a departed employee? → It must be preserved (soft delete + audit).

### Step 6 — Anticipate risks and dependencies

A User Story that doesn't anticipate its risks and dependencies creates surprises mid-sprint. The following were identified:

- **Technical dependencies**: other modules that must exist first (areas, audit log, email API)
- **Adoption risks**: HR team resistance to change
- **Legal risks**: compliance with Argentina's Personal Data Protection Law (Law 25.326)

### Step 7 — Validate against the company's quality standards

Once the story was drafted, a gap analysis was performed against the company's active certifications: **ISO/IEC 27001:2022** (Information Security) and **ISO 9001:2015** (Quality Management). This step is detailed in section 7.

## 4. Anatomy of the User Story

Each section of the story serves a specific purpose. This table explains the role of each one:

| Section | Purpose | Primary audience |
|---|---|---|
| **Metadata** (ID, epic, sprint, SP) | Traceability and backlog planning | Scrum Master, PM |
| **User Story** | Defines value from the user's perspective | Entire team |
| **Context** | Justifies *why* this feature is needed | Stakeholders, new team members |
| **Personas / Roles** | Prevents designing for only one type of user | UX, Dev, PM |
| **Data to manage** | Defines the data model at a high level | Backend, DBA |
| **Acceptance Criteria** | Verifiable conditions to consider the story "done" | QA, Dev |
| **Business Rules** | Cross-cutting restrictions and behaviors | Dev, QA, Legal |
| **User Flow** | Visualizes the happy path sequence | UX, Frontend Dev |
| **Definition of Done** | Quality checklist agreed upon by the team | Entire team |
| **Assumptions & Dependencies** | Prevents blockers and false assumptions in the sprint | PM, Tech Lead |
| **Risks** | Anticipates problems before they occur | PM, Stakeholders |
| **ISO Compliance** | Maps applicable regulatory controls for audit traceability | Quality Manager, ISO Auditor |
| **Quality Metrics** | Defines measurable indicators to validate success in production | PM, Quality Manager |
| **Out of Scope** | Defines explicit boundaries to prevent scope creep | PM, Stakeholders |

## 5. Frameworks and Tools Used

### 5.1 Connextra Template
The standard User Story format ("As / I want / So that") proposed by Rachel Davies at Connextra (2001). It is the universal starting point in agile methodologies.

### 5.2 MoSCoW Prioritization
A framework for prioritizing features based on their business impact:

| Category | Meaning | Application in this US |
|---|---|---|
| **Must Have** | Critical — the system doesn't work without this | Employee creation and deactivation |
| **Should Have** | Important, but the system functions without it | Excel export |
| **Could Have** | Desirable if time allows | Advanced filters |
| **Won't Have** | Out of scope for this iteration | Org chart, vacation module |

This US was classified as **Must Have** because other system modules depend on it.

### 5.3 Given / When / Then (BDD Format)
Also known as **BDD (Behavior-Driven Development)**, this format for acceptance criteria was popularized by Dan North. Its advantage is that it can be read by both business stakeholders and developers/testers, and can be directly translated into automated tests.

```
Given   [an initial context / precondition]
When    [an action is performed]
Then    [an observable and verifiable result occurs]
```

### 5.4 Story Points (Relative Estimation)
**Story Points** are not hours. They are a relative measure of effort, complexity, and uncertainty. The **Fibonacci sequence** (1, 2, 3, 5, 8, 13, 21) is commonly used because differences between larger numbers are harder to estimate precisely.

This story was initially estimated at **8 SP** and later re-estimated to **13 SP** after incorporating the acceptance criteria and business rules derived from the ISO compliance analysis.

### 5.5 Definition of Done (DoD)
The DoD is a team agreement that defines what conditions any story must meet to be considered complete. It is not specific to this story — it is a team-wide standard. Including the DoD in the story ensures that everyone involved has clarity on what "finished" means.

In this story, the DoD was expanded to include security verifications (encryption) and process verifications (ISO traceability) as a result of the compliance analysis.

### 5.6 INVEST Criteria
Every good User Story should meet the **INVEST** criteria:

| Letter | Criterion | Does this US comply? |
|---|---|---|
| **I** · Independent | As independent from other stories as possible | Has declared dependencies but doesn't block other US |
| **N** · Negotiable | Details are open to discussion, not a contract | Fields and rules are a starting point |
| **V** · Valuable | Delivers clear value to the user | Replaces an error-prone manual process |
| **E** · Estimable | The team can estimate the effort | 13 SP agreed upon |
| **S** · Small | Small enough to complete in a sprint | Borderline — could be split if the team decides it's necessary |
| **T** · Testable | Can be objectively verified | Acceptance criteria in BDD format |

## 6. Embedded Business Logic

This section explains the non-trivial business decisions implicit in the story that a reader without context might not understand.

### 6.1 Soft Delete vs. Physical Deletion

**What is it?**
When an employee leaves the company, their record is **not deleted** from the database. Instead, it is marked as "Inactive" and given an end date and a reason. The record continues to exist and can be queried.

**Why?**
- **Audit trail**: companies are required to maintain employment history.
- **Legal compliance**: Argentina's Employment Contract Law requires certain data to be retained for years.
- **Referential integrity**: if the employee was linked to other records (approvals, reports, tasks), deleting their record would break those relationships.
- **Rehire scenarios**: if the employee returns, their full history can be restored by reactivating the record.

### 6.2 Auto-generated, Non-editable Employee ID

**What is it?**
The Employee ID (called *legajo* in Argentina) is the unique internal number that identifies an employee within the company.

**Why auto-generated?**
- Prevents duplicates and human errors.
- Guarantees consistent, sequential numbering.
- Simplifies the user experience (HR Admin doesn't need to decide the number).

**Why non-editable?**
Once assigned, the Employee ID may be referenced in contracts, pay stubs, access control systems, and other documents. Allowing edits would create serious inconsistencies across those references.

### 6.3 Auto-generated Corporate Email

**What is it?**
When a new employee is created, the system automatically generates a corporate email following the pattern `firstname.lastname@company.com`.

**Why automate it?**
- Ensures consistency in the format of internal email addresses.
- Removes a manual step from the onboarding process.
- Can integrate with the email provider (Google Workspace, Microsoft 365) to automatically provision the account.

**Edge cases to consider** (not covered in this version):
- Two employees with the same first and last name → the system should append a number (`john.smith2@company.com`)
- Employees with compound names or surnames with spaces → requires a normalization policy

### 6.4 Role-Based Permissions

The story defines four distinct access levels:

| Role | Create | Edit | Deactivate | View |
|---|---|---|---|---|
| Superadmin | Yes | Yes | Yes | Yes |
| HR Admin | Yes | Yes | Yes | Yes (all) |
| Area Manager | No | No | No | Yes (own team only) |
| Employee | No | No | No | Yes (own data only) |

This reflects the **principle of least privilege**: each user accesses only what they need to perform their role.

### 6.5 Audit Log

**What is it?**
An immutable record of every change made to any employee record. Each log entry contains:
- User who made the change
- Exact date and time
- Modified field
- Previous value
- New value

**Why does it matter?**
- Enables the company to respond to labor or legal inspections
- Detects unauthorized changes
- Is explicitly required by ISO/IEC 27001:2022 (control 8.15) and by Argentina's Law 25.326
- Logs must be retained for a minimum of 5 years and cannot be modified or deleted by any role (BR-08)

### 6.6 Law 25.326 — Personal Data Protection

This Argentine law (comparable in spirit to the European GDPR) regulates how companies must handle personal data. In the context of this module, it means:

- Employees have the right to access their own data
- Certain fields (such as `dni_cuil_cifrado`) cannot be freely shared across roles
- Data must be protected with appropriate security measures (encryption at rest)
- Access to sensitive data must be justified and recorded

### 6.7 Access Revocation on Deactivation

When an employee is deactivated, the system does not only change their status — it also automatically revokes their access to the corporate email account and notifies the IT team. This addresses a real security risk: a former employee with active access to internal systems represents a vulnerability. This behavior is covered by AC-08 and is required by ISO/IEC 27001:2022 control 5.18.

### 6.8 Encryption and Masking of Sensitive Data

The `dni_cuil_cifrado` field is stored encrypted at rest (AES-256 or equivalent) and displayed masked in the interface for roles without explicit permission. This addresses two simultaneous needs:

- **Security**: protecting sensitive data in the event of a database breach (ISO/IEC 27001:2022 control 8.24)
- **Privacy**: an area manager consulting their team does not need to see each person's full ID number

## 7. ISO Compliance Framework — Requirements and Coverage

This section describes what the applicable ISO standards require from a module that manages personal and employment data, and how each requirement is covered within this User Story through acceptance criteria, business rules, Definition of Done items, or satellite technical stories.

### 7.1 Regulatory Context

The company operates under two certifications issued by IRAM with international IQNet recognition:

| Standard | Current version | Scope |
|---|---|---|
| **ISO/IEC 27001** | 2022 *(with 2024 amendment)* | Information Security — risk management, access controls, cryptography, auditing |
| **ISO 9001** | 2015 | Quality Management — processes, traceability, customer satisfaction, continual improvement |

> **Note on versions:** There is no ISO 27001:2024 or ISO 9001:2024 as independent editions of these standards. The "2024" shown on the IRAM certificate corresponds to the date of issuance or audit of the certificate, not a new version of the standard. The next revision of ISO 9001 is expected for late 2026 (still in draft); for ISO 27001 no revision date has been announced.

### 7.2 ISO/IEC 27001:2022 — Information Security

ISO/IEC 27001 is the international standard for information security management. It establishes the requirements for implementing, maintaining, and continually improving an Information Security Management System (ISMS). Its Annex A defines 93 controls organized into four domains: organizational, people, physical, and technological.

For a module that manages personal and employment data, the most relevant controls are the following:

**Control 5.12 — Classification of information**

This control requires that information be classified according to its sensitivity level and that access and handling be managed accordingly. In this story, the `dni_cuil_cifrado` field is identified as sensitive data: it cannot be exported or displayed in plain text by roles without explicit permission. This is implemented through AC-07 (role-restricted export) and AC-09 (UI masking).

**Control 5.15 — Access control**

Requires that access to information be restricted based on each role's needs. This story implements four distinct access levels: Superadmin, HR Admin, Area Manager, and Employee. Each role has specific permissions for creation, editing, deactivation, and viewing, defined in BR-01 and BR-07.

**Control 5.18 — Access rights**

Establishes that access rights must be revoked when no longer needed, particularly upon termination of employment. AC-08 ensures that when an employee is deactivated, the system automatically revokes their corporate email access, notifies the IT team, and records the action in the audit log.

**Control 5.31 — Legal and contractual requirements**

Requires identifying and complying with applicable legal obligations related to data handling. The DoD includes formal verification with Legal of compliance with Argentina's Personal Data Protection Law (Law 25.326), which regulates the handling of personal data in a manner analogous to the European GDPR.

**Control 8.5 — Secure authentication**

Requires that access to systems be protected through secure authentication mechanisms. This module depends on the centralized authentication system (SSO) already implemented, documented as an assumption in the story. Its implementation and compliance are the responsibility of the authentication module.

**Control 8.10 — Deletion of information**

Establishes that information must be managed so it cannot be recovered in an unauthorized manner after deletion. This module applies the soft delete principle (BR-02): no employee record can be physically deleted from the system, ensuring permanent integrity and traceability.

**Control 8.13 — Information backup**

Requires that information be backed up regularly and that backups be verifiable. This requirement is covered by the IT/DevOps infrastructure backup policy, documented as out of scope for this User Story.

**Control 8.15 — Logging**

Requires that relevant system events be recorded and that those records be protected against modification or deletion. BR-06 establishes that every modification is logged in the audit trail. BR-08 adds that those logs must be retained for a minimum of 5 years and cannot be altered by any role.

**Control 8.24 — Use of cryptography**

Requires that sensitive stored information be protected through encryption. BR-09 establishes that the `dni_cuil_cifrado` field is stored encrypted at rest using AES-256 or equivalent. The technical implementation of this encryption is the responsibility of the satellite story US-001-SEC.

### 7.3 ISO 9001:2015 — Quality Management

ISO 9001 is the international standard for quality management systems (QMS). Its core focus is continual improvement, customer orientation, and process- and risk-based management. Unlike ISO 27001, it does not prescribe specific technical controls but rather principles and clauses that must be materialized in the organization's processes.

For a User Story in an agile environment certified under ISO 9001, the most relevant clauses are:

**Clause 6.1 — Actions to address risks and opportunities**

Requires the organization to identify risks that may affect the achievement of intended results and define mitigation actions. This story includes a Risks section with five identified risks, their impact level, and corresponding mitigation actions.

**Clause 6.3 — Planning of changes**

Establishes that changes to the management system must be carried out in a planned manner. In the context of this story, any modification to scope, criteria, or rules is documented in the design process, with Story Points updated and reasons recorded.

**Clause 8.1 — Operational planning and control**

Requires that the processes needed to meet requirements be planned, controlled, and documented. This story covers this requirement through the documentation of assumptions, dependencies, user flow, and verifiable acceptance criteria.

**Clause 8.5.1 — Control of production and service provision**

Establishes that controlled conditions for service delivery must include acceptance criteria and evidence of conformity. The Definition of Done for this story includes test coverage of 80% or greater, staging validation, and Product Manager approval before closure.

**Clause 8.5.2 — Traceability**

Requires that the product or service can be identified and traced throughout its lifecycle. This story is identified with a unique ID (US-001), associated with an epic and sprint, and its dependencies are referenced by name. Acceptance criteria are numbered and traceable to the ISO controls that originated them.

**Clause 8.6 — Release of products and services**

Establishes that a product must not be released until planned arrangements have been satisfactorily completed. The DoD for this story includes formal approval from the Quality Manager as a closing condition, in addition to Product Manager validation.

**Clause 9.1.1 — Monitoring, measurement, analysis and evaluation**

Requires the organization to determine what needs to be monitored and measured, and with what methods. This story includes a Quality Metrics section with five measurable indicators — creation error rate, average creation time, list response time, successful revocation rate, and user satisfaction — each with a defined acceptable threshold and a 30-day post go-live measurement period.

**Clause 9.1.2 — Customer satisfaction**

Requires monitoring customer perception regarding the fulfillment of their requirements. In this module, the internal "customer" is the HR team. The user satisfaction metric (post go-live survey, minimum threshold of 4.0 out of 5) covers this requirement directly and measurably.

## 8. Design Decisions

These are the conscious decisions made while writing the story, along with the reasoning behind each.

### Why include an "Out of Scope" section?

**Scope creep** — uncontrolled expansion of a project's scope — is one of the leading causes of software delivery delays. Explicitly defining what a story does *not* include is just as valuable as defining what it does. A stakeholder who reads "vacation management: out of scope" knows to look for a separate story for that, rather than assuming it's bundled here.

### Why separate Business Rules from Acceptance Criteria?

**Acceptance Criteria** describe behavior from the user's perspective ("when I do X, I see Y"). **Business Rules** are restrictions and policies that apply across multiple flows and that the system must always respect, regardless of the specific scenario. Mixing them produces confusion and redundant, overlapping criteria.

### Why include risks in a User Story?

A User Story that reaches the sprint without anticipating its risks can get blocked mid-development. Identifying risks during refinement allows the team to take preventive action (e.g., coordinate with Legal before writing a single line of code) rather than reactive action (discovering the problem when the story is already in progress).

### Why organize employee data into three groups?

Separating data into **Personal / Employment / System** has both technical and UX implications:

- In the **frontend**, it allows the form to be organized into logical tabs that reduce cognitive load for the user.
- In the **backend**, it suggests a separation of concerns (the personal data table could potentially be reused if the company also manages clients or other actors).
- In terms of **permissions**, some fields (e.g., `dni_cuil_cifrado`) are more sensitive than others and require encryption at rest and UI masking.

### Why not assign IDs to dependency stories?

In a real backlog, the IDs of other stories only exist once those stories have been created and prioritized. Inventing arbitrary IDs (such as US-002, US-010) creates confusion: it implies a structured backlog with 10+ stories when in reality those stories have not yet been defined. The decision was to reference dependencies by descriptive name and mark them as "pending creation", which is the honest state.

### Why create a satellite technical story (US-001-SEC) instead of including everything here?

Encryption at rest is architecture and backend work that can be estimated, planned, and assigned independently of the functional CRUD flow. Including it in this story would unnecessarily inflate its scope and complicate sprint planning. The satellite story allows both tracks to advance in parallel if the team decides to do so.

## 9. Glossary

| Term | Definition |
|---|---|
| **ABM** | Alta, Baja y Modificación. Spanish-language equivalent of CRUD, used in Latin American software development. |
| **AES-256** | Advanced Encryption Standard with a 256-bit key. A widely used symmetric encryption standard for protecting data at rest. |
| **Audit log** | An immutable record of all changes made to a data record, including who made the change, when, which field was modified, and the old and new values. |
| **Backlog** | A prioritized list of all features, improvements, and fixes pending for a product. |
| **BDD** | Behavior-Driven Development. A development methodology that defines system behavior in natural language using the Given/When/Then format. |
| **CRUD** | Create, Read, Update, Delete. The four basic operations performed on data records in a system. |
| **DoD** | Definition of Done. A team-agreed quality checklist that every story must satisfy before it can be closed. |
| **Edge case** | A boundary or unusual situation that may reveal unexpected system behavior. |
| **Employee ID (Legajo)** | A unique internal number assigned to identify an employee within a company. The term *legajo* is the Argentine Spanish equivalent. |
| **Encryption at rest** | A security technique that protects data stored in a database or file system using cryptographic algorithms. Data is only readable by those with the key. |
| **Epic** | A large unit of work that groups several related User Stories under a common objective. |
| **Gap analysis** | The process of comparing the current state of an artifact against a standard or requirement to identify what is missing. |
| **HRMS / HCM** | Human Resource Management System / Human Capital Management. Software systems for managing employee data and HR processes. |
| **INVEST** | Acronym for criteria to evaluate the quality of a User Story: Independent, Negotiable, Valuable, Estimable, Small, Testable. |
| **ISO/IEC 27001:2022** | International standard for information security management systems (ISMS). The 2022 edition is currently in force; it received a minor amendment in 2024. |
| **ISO 9001:2015** | International standard for quality management systems (QMS). Currently in force; a new revision (ISO 9001:2026) is in draft and expected in late 2026. |
| **Least privilege** | A security principle stating that each user should have access only to the resources necessary to perform their role. |
| **Masking** | A UI technique that partially hides a sensitive data field on screen (e.g., `••••••••123`) to protect privacy without preventing identification. |
| **MoSCoW** | A prioritization framework: Must Have, Should Have, Could Have, Won't Have. |
| **Payroll** | The system or process used to calculate and distribute employee salaries and wages. |
| **Satellite story** | A technical User Story derived from a primary story, covering a specific aspect (typically architecture or security) that merits independent estimation and planning. |
| **Scope creep** | Uncontrolled expansion of a project's scope, typically caused by adding unplanned features. |
| **Soft delete** | Deactivating a record by changing its status (e.g., to "Inactive") rather than physically removing it from the database. |
| **Sprint** | A fixed time period (typically 1–4 weeks) during which a Scrum team commits to completing a set of stories. |
| **Story Points** | A relative unit of measure for the effort, complexity, and uncertainty of a User Story. They do not equal hours. |
| **User Story** | A short description of a feature from the end user's perspective, expressing what they need and why. |
